# CMS Data Model Handoff

Use BaseHub as the hosted CMS for v1. The public site and government export should both read from the same content model.

## Modeling Decisions

- V1 is built for one developer and one active investment, but the schema should allow multiple investments later.
- The content hierarchy is `Developer -> Investment -> Unit -> Price history/components`.
- Investment means one development project, for example an estate of houses or one condo/apartment project.
- Listing/unit means one house or one apartment inside an investment.
- The site should keep sold/reserved units visible with status unless the developer's lawyer confirms a different rule.
- Price changes should be append-style records, not normal in-place edits of previous price records.
- Public listing pages should show price history.
- Contact form is out of scope for v1. Show phone, email, company data, and sales office details instead.
- Prospectus fields should be present because the checklist expects the general prospectus information to be published on the website, and government guidance recommends including the URL where it is available.

## Document Types

### Developer

Company-level information shared across the website and CSV export.

Recommended fields:

- `name`
- `slug`
- `legalForm`
- `krsNumber`
- `ceidgNumber`
- `nip`
- `regon`
- `phone`
- `email`
- `fax`
- `websiteUrl`
- `registeredAddress`
- `salesOfficeAddress`
- `additionalSalesLocations`
- `buyerContactMethod`
- `logo`
- `socialLinks`

Addresses should be structured, not free-text only:

- `voivodeship`
- `county`
- `municipality`
- `city`
- `street`
- `buildingNumber`
- `unitNumber`
- `postalCode`

### Investment

One real-estate development project.

Recommended fields:

- `name`
- `slug`
- `developer`
- `summary`
- `description`
- `locationAddress`
- `heroImage`
- `gallery`
- `prospectusUrl`
- `prospectusFile`
- `salesStatus`: `active` or `ended`
- `salesEndedAt`
- `salesEndedNote`
- `governmentDatasetExtIdent`
- `governmentDatasetTitle`
- `governmentDatasetDescription`
- `isReportedToGovernment`

For v1, build the UI around one active investment. Do not hard-code this in the core model or export logic.

When `salesStatus` is `ended`, the XML generator should append a clear end-of-sale sentence to the dataset description, for example:

```text
Sprzedaż inwestycji {investment.name} zakończyła się z dniem {salesEndedAt}.
```

### Unit

One house/apartment listed inside an investment.

Recommended fields:

- `investment`
- `unitNumber`
- `slug`
- `propertyType`: `dom jednorodzinny` or `lokal mieszkalny`
- `status`: `available`, `reserved`, `sold`, `hidden`
- `usableAreaM2`
- `rooms`
- `floor`
- `plotAreaM2`
- `shortDescription`
- `description`
- `mainImage`
- `gallery`
- `features`
- `priceHistory`
- `propertyParts`
- `appurtenantRooms`
- `useRights`
- `otherFees`
- `includedInPriceNotes`
- `governmentReportingEnabled`

Use `hidden` only for editorial/public-site hiding. Do not silently exclude from government export without a deliberate rule.

### Price History Entry

Each business price change should create a new entry.

Recommended fields:

- `validFrom`
- `pricePerM2`
- `baseTotalPrice`
- `totalPriceWithComponents`
- `vatIncluded`
- `note`
- `correctionOfPreviousEntry`

Exporter behavior:

- Current price is the latest entry with `validFrom <= exportDate`.
- Public pages display all entries sorted newest-first or oldest-first, but keep the dates visible.
- If correcting a typo, use `correctionOfPreviousEntry` and notes rather than hiding the existence of prior published data.

### Price Components

The government CSV expects separate categories. Model them separately even if the public UI presents them simply.

`propertyParts`:

- `type`, for example `Miejsce postojowe`
- `identifier`
- `price`
- `validFrom`
- `required`

`appurtenantRooms`:

- `type`, for example `Komórka lokatorska`
- `identifier`
- `price`
- `validFrom`
- `required`

`useRights`:

- `description`
- `value`
- `validFrom`

`otherFees`:

- `description`
- `value`
- `validFrom`

Use `X` in CSV output where a category is not applicable.

## Editing Workflow

The developer should edit listings in BaseHub:

1. Update descriptive information and images directly.
2. For price changes, add a new dated price history entry.
3. For sold/reserved changes, update `status`.
4. For investment ending, set `salesStatus=ended` and `salesEndedAt`.

BaseHub publish should trigger a public site rebuild so visible listing pages update quickly. The official daily CSV/XML/MD5 snapshot is generated by scheduled automation.

## Validation Rules

Future agents should add validation in BaseHub model and exporter code:

- Developer legal/contact fields required before launch.
- Investment address, sales status, and prospectus URL/file required before launch.
- Unit must have `unitNumber`, property type, status, area, and at least one price history entry before it is reportable.
- Price dates must be valid dates and should not duplicate the same `validFrom` for one unit unless explicitly marked as correction.
- Government ext identifiers must be stable, ASCII, max 36 characters, and unique within one XML file.
- Public URLs used in XML must be HTTPS.

## Local Fixture

Create a local fixture so export tests do not depend on BaseHub credentials. Suggested path:

```text
fixtures/sample-content.json
```

It should include:

- one developer,
- one active investment,
- at least three units,
- at least one unit with a price change,
- one parking/storage component example,
- one `X`/not-applicable case.

Tests should generate CSV/XML/MD5 from this fixture and compare against stable expected outputs.
